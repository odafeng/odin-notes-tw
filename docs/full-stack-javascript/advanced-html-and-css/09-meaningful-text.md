# 有意義的文字

> 改寫自 The Odin Project：[Meaningful Text](https://www.theodinproject.com/lessons/node-path-advanced-html-and-css-meaningful-text)
> ｜Full Stack JavaScript › Advanced HTML and CSS › Accessibility

## 核心概念

「有意義的文字」（meaningful text）這個概念其實很直觀：當使用者讀到一段文字，或由輔助科技（assistive technology，例如螢幕閱讀器 screen reader）把它唸出來時，即使完全沒有周圍的情境（context），他也應該能立刻理解這段文字是什麼意思。文字寫得不夠有意義，會影響所有使用者，但對倚賴輔助科技的人影響最深。

為什麼影響最深？因為「看得到畫面」的使用者，往往能靠版面、顏色、圖示、位置這些視覺線索補足缺漏的資訊。舉例來說，一個藍色、有底線、旁邊還畫了箭頭的「這裡」，明眼人一看就知道是連結、大概也猜得到用途。但螢幕閱讀器使用者拿不到這些視覺線索，他們拿到的就只有你寫下的那幾個字。如果那幾個字本身講不清楚，資訊就真的遺失了。

這一課會帶你看三個最常見、也最容易寫出「沒有意義的文字」的地方：**連結（links）**、**表單（forms）**，以及**圖片的替代文字（alternative text，即 `alt` 屬性）**。掌握這三處，你就能在日常開發中大幅提升頁面的可及性（accessibility，簡稱 a11y）。

### 連結：讓連結文字自己說清楚要去哪

先看兩個寫法不同、但功能一樣的連結：

```html
<!-- 範例 1：「這裡」是哪裡？ -->
<a href="...">點這裡</a>開始你的網頁開發職涯！

<!-- 範例 2：我超愛那個地方！ -->
造訪 <a href="...">The Odin Project</a> 開始你的網頁開發職涯！
```

對看得到畫面的使用者來說，範例 1 讀起來完全沒問題，因為他能連同前後整句話一起讀。但螢幕閱讀器除了能靠地標（landmark）和標題（heading）在頁面上跳著導覽之外，通常還能讓使用者「只在某一類元素之間跳轉」，例如：一個接一個地在所有連結之間移動。這是一種很常見的瀏覽方式，因為連結往往就是使用者想採取的下一步動作。

問題來了：當使用者用這種方式逐一跳過連結時，範例 1 被唸出來的只有一句「點這裡，連結」（Click here, link）。這裡到底是哪裡？脫離了前後文，這個連結完全沒有意義。更糟的是，如果頁面上有好幾個連結的文字都是「點這裡」，使用者就會被連續唸到好幾次一模一樣的「點這裡、點這裡、點這裡」，根本無從分辨。

範例 2 就不同了。它不只在句子裡讀起來通順（對所有使用者都友善），連被單獨唸出來、脫離情境時也依然清楚：「The Odin Project，連結」。使用者一聽就知道這個連結會把他帶去哪裡。

當你在頁面上加連結時，有三條規則值得養成習慣：

1. **連結的文字內容要能指出它會導向哪裡，而且要簡潔**（大約 100 個字元以內）。避免用「點這裡」「這一頁」「更多」這種空泛的詞。理想的連結文字，往往就是目標頁面、文章或影片的標題。
2. **如果連結會開啟或下載一個檔案，就在文字裡標明檔案類型，以及檔案大小。** 這樣使用者在點下去之前，就知道自己即將面對的是什麼、要不要在目前的網路環境下載。
3. **如果連結會用 `target="_blank"` 自動在新分頁或新視窗開啟，你應該用某種方式告知使用者。** 因為「頁面突然跳到新分頁」對倚賴螢幕閱讀器或鍵盤導覽的人來說相當困惑，尤其是當他們想按「上一頁」卻發現按不回去的時候。

把後兩條套進實際寫法，會長這樣：

```html
<!-- 現在使用者知道這個連結會開啟或下載一個 PDF 檔案，而且大約 1MB -->
<a href="...">2021 年註冊統計（PDF，1MB）</a>

<!-- 現在使用者知道這個連結會在新分頁開啟 -->
<a href="...">GitHub（在新分頁開啟）</a>
```

下次要放連結時，可以試試一個簡單的自我檢查：**把連結的文字內容單獨唸出來聽聽看。** 光聽這幾個字，你能合理判斷它會把你帶去哪裡嗎？你知道它會不會自動開新分頁、會不會跳出下載對話框嗎？如果到目前為止你已經在練習用螢幕閱讀器測試網頁，那最好的驗證方式，就是直接用螢幕閱讀器實際聽一次。

### 表單：把錯誤訊息與說明寫得有意義

填寫或送出表單（form）時遇到錯誤，很容易讓人挫折。如果錯誤訊息寫得有意義，雖然不會讓填表單變得多好玩，但至少能讓整個過程「沒那麼氣人」。我們來看三個從「毫無幫助」到「非常有幫助」的錯誤訊息：

```html
<!-- 範例 1：蛤？ -->
<div class="input-error">錯誤：輸入無效。</div>

<!-- 範例 2：這比較合理了 -->
<div class="input-error">錯誤：Email 無效。</div>

<!-- 範例 3：更好！ -->
<div class="input-error">
  錯誤：「JohnSmith@@test.com」不是有效的 Email。有效範例：example@yourdomain.com。
</div>
```

範例 1 的問題在於：就算你「猜得到」是哪個欄位出錯（實際上常常猜不到），這句話也沒提供任何有意義的資訊——**哪個輸入無效？為什麼無效？該怎麼修正？** 一個問題都沒回答。再想像一下，這種訊息對輔助科技使用者有多沒用：他們可能根本看不到錯誤被渲染（render）在畫面上的哪個位置，耳朵裡只聽到孤零零一句「輸入無效」。

範例 2 明確告訴你是「Email」這個欄位無效，你至少知道該回頭去看哪裡了。對很多情況來說，這樣或許就夠用了；但要記得，它其實還是有點模糊——我們並不知道 Email「為什麼」無效。

範例 3 又更進一步。它不只指出是哪個欄位出錯，還告訴你該怎麼修正（附上一個正確格式的範例）。這就帶出一條通則：**當你要通知使用者表單有錯時，應該讓他知道「是哪個輸入造成錯誤」，並且在可能的情況下，告訴他「該如何修正」或「為什麼會發生這個錯誤」。**

除了錯誤訊息，表單裡另一種需要「有意義文字」的地方是**操作說明（instructions）**，例如密碼欄位列出的規則：「必須包含至少一個大寫字母和一個數字…」。放這些說明時有個原則要注意——

- **針對「單一欄位」的說明**（例如這個密碼欄位的格式要求），應該放在該欄位「旁邊」，讓使用者在填那一格時就能就近讀到。
- **橫跨「整個表單」的說明**（例如標示哪些欄位是必填），則有兩種好做法：放在表單的最上方（「* 表示必填欄位」），或是放在該欄位或其標籤（label）旁邊（「姓名（必填）」）。

這裡的重點是「就近」與「及時」：說明要出現在使用者需要它的當下與位置，而不是藏在別處，逼使用者自己去把資訊拼湊起來。

### 替代文字：連「沒有文字」也可以是有意義的

到這裡你應該已經很熟悉 `img` 元素上的 `alt` 屬性了。不管熟不熟，先來考你一題：下面兩個範例，哪一個是「合法（valid）」的寫法？

```html
<!-- 範例 1 -->
<img src="..." alt="" />

<!-- 範例 2 -->
<img src="..." alt="Odin" />
```

信不信由你——**兩個都合法。**

範例 1 雖然沒有任何有意義的文字（或者說，它是一種「有意義地『不放』文字」），但你還是得理解它的重要性。當一張圖**純粹是裝飾用途**，或這張圖對使用者來說根本不重要時，你通常「不希望」輔助科技的使用者被告知它的存在——不然只是徒增干擾。這種情況下，你應該**永遠**把 `alt` 屬性的值設成空字串，就像範例 1 那樣。這個空字串也被稱為「null 值」（null value，注意別跟 JavaScript 的資料型別 `null` 搞混，這裡指的是「空的替代文字」）。

關鍵在於：**「設成空字串」和「乾脆不寫 `alt`」是兩回事。** 如果你直接把 `alt` 屬性省略掉，圖片的存在仍然可能被螢幕閱讀器宣告出來，反而讓使用者困惑——尤其當圖片的檔名是一串隨機的英數字（例如 `IMG_8F3K92.png`）時，使用者可能會聽到這串亂碼，卻完全不知道那是什麼。所以，「裝飾性圖片」的正確做法是**明確地寫上 `alt=""`**，主動告訴輔助科技「這張圖可以略過」。

至於範例 2，螢幕閱讀器會唸出「Odin，圖片」（Odin, graphic），讓使用者知道這裡有一張圖、而且它是關於「Odin」的。不過，一張圖的替代文字到底該寫什麼，最終會取決於**這張圖的功能，以及它周圍的情境**——同一張圖，放在不同位置、扮演不同角色，適合的 `alt` 內容可能完全不同。這也是為什麼替代文字沒有一體適用的公式，而需要你針對每張圖去判斷：如果這張圖消失了，使用者會漏掉哪些「靠文字才能補回來」的資訊？把那些資訊寫進 `alt`，就對了。

### 一個貫穿全課的心法

把三個主題收攏起來，其實是同一件事：**不要倚賴使用者「看得到」的東西來傳遞意義。** 連結不要靠周圍句子才說得清、錯誤不要靠畫面位置才看得懂、圖片不要靠視覺本身才有價值。把意義直接寫進文字裡，你的介面就能對更多人成立。一個很實用的隨手測試是：把某段內容（連結、錯誤、`alt`）單獨抽出來、脫離畫面地讀一次，如果它依然講得清楚，你就寫對了。

## 程式碼範例

下面是一個把三個原則整合在一起的小片段，可以直接貼進 HTML 檔案用瀏覽器開啟觀察：

```html
<!-- 有意義的連結：文字指出目的地，並標明檔案類型與大小 -->
<p>
  想深入了解，請閱讀
  <a href="/report.pdf">2021 年年度報告（PDF，2MB）</a>，
  或造訪
  <a href="https://www.theodinproject.com" target="_blank">
    The Odin Project（在新分頁開啟）
  </a>。
</p>

<!-- 有意義的表單：說明就近放，錯誤訊息指出欄位並示範修正 -->
<form>
  <p>* 表示必填欄位</p>

  <label for="email">Email（必填）</label>
  <input id="email" type="email" name="email" />
  <!-- 這段錯誤訊息會由驗證邏輯在需要時顯示 -->
  <div class="input-error">
    錯誤：「JohnSmith@@test.com」不是有效的 Email。有效範例：example@yourdomain.com。
  </div>

  <label for="pwd">密碼</label>
  <!-- 針對單一欄位的說明，就近放在欄位旁 -->
  <small id="pwd-hint">必須包含至少一個大寫字母和一個數字。</small>
  <input id="pwd" type="password" name="pwd" aria-describedby="pwd-hint" />
</form>

<!-- 有意義的替代文字：資訊性圖片描述內容，裝飾性圖片用空字串 -->
<img src="odin-logo.png" alt="Odin 專案的標誌" />
<img src="decorative-divider.png" alt="" />
```

## 常見陷阱

!!! warning "把 `alt=\"\"` 和「省略 alt」當成一樣"
    兩者結果不同。裝飾性圖片要**明確寫** `alt=""`，螢幕閱讀器才會直接略過它；如果你乾脆不寫 `alt`，圖片的存在仍可能被宣告，甚至唸出無意義的檔名，反而干擾使用者。只有「這張圖對使用者不重要」時才用空字串；只要圖片承載了資訊，就必須寫出有意義的描述。

!!! warning "用「點這裡」「更多」當連結文字"
    螢幕閱讀器使用者常會逐一跳過所有連結，此時只會聽到連結本身的文字。「點這裡」脫離情境後毫無意義，多個相同文字的連結更會讓人完全無法分辨。連結文字要能自己說清楚目的地。

!!! warning "錯誤訊息只說「輸入無效」"
    看不到畫面的使用者無法從位置判斷是哪一欄出錯。錯誤訊息至少要指出「哪個欄位」，最好再說明「為什麼錯」或「怎麼修正」，例如附上一個正確格式的範例。

!!! warning "把表單說明藏在遠處"
    針對單一欄位的規則（如密碼格式）要就近放在該欄位旁；跨表單的說明（如必填標示）放在表單頂端或標籤旁。別讓使用者自己去別處把資訊拼湊起來。

## 練習

以下改寫自原文的 Assignment，建議依序完成：

1. 閱讀 WebAIM 的〈Alternative Text〉一文，理解「該不該加替代文字、該怎麼加」是如何取決於**圖片的功能**與**周圍情境**的。試著在自己過去的專案裡找幾張圖，逐一問自己：這張圖是資訊性的還是裝飾性的？如果是資訊性的，缺了它使用者會漏掉什麼？據此寫出（或修正）它的 `alt`。
2. 閱讀 WebAIM 的〈Usable and Accessible Form Validation and Error Recovery〉一文，了解提供表單錯誤的不同做法，以及每種做法的優缺點。接著回頭檢視你寫過的表單，把其中模糊的錯誤訊息改寫成「指出欄位 + 說明如何修正」的版本。
3. 如果你已經安裝並會操作螢幕閱讀器，實際用它逐一跳過頁面上的連結、觸發一次表單錯誤、瀏覽一張圖片，親耳確認你寫的文字是否真的有意義。
