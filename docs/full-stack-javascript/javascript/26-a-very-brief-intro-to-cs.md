# 計算機科學極簡入門

> 改寫自 The Odin Project：[A Very Brief Intro to CS](https://www.theodinproject.com/lessons/javascript-a-very-brief-intro-to-cs)

## 核心概念

到目前為止，你已經能寫出可以運作的程式，甚至能做出像樣的網站了。但「能運作」和「寫得好」之間，其實隔著一段距離。就像一個人能寫出文法正確的英文句子，不代表他明天就能去《紐約時報》當編輯一樣：把問題解出來只是起點，把問題解得**優雅**（elegant）才是這一系列課程要訓練的能力。

在真實的工程世界裡，解決一個問題往往有兩種路線。一種是**暴力解**（brute force）：不管三七二十一，用最直覺、最粗魯的方式把答案硬湊出來。另一種是**優雅解**：先理解問題的本質，選對工具與策略，用更少的步驟、更少的資源把同一件事做好。當你的資料量很小時，這兩種解法感覺不出差別；但一旦你的網站爆紅、資料量成長到數百萬筆，暴力解可能會讓伺服器卡到動不了，而優雅解卻依然順暢。這就是為什麼我們要回頭補一點**計算機科學**（computer science，簡稱 CS）的基礎——這些概念之所以是「基礎」，是有原因的。

這一課的目標很單純，只有兩個關鍵詞：**algorithm（演算法）** 與 **pseudocode（虛擬碼）**。我們刻意停留在實用面，不鑽進太深的理論。

### 什麼是 algorithm（演算法）

**algorithm（演算法）** 是「一組有限、精確、可執行的指令序列，用來解決某一類問題或完成某個計算」。這個定義聽起來很學術，但拆開來看其實很貼近生活。一份食譜是演算法：先做什麼、再做什麼、放多少鹽、烤幾分鐘，照著做就能得到蛋糕。組裝家具的說明書、教你怎麼從家裡走到車站的路線指示，本質上都是演算法。

演算法有幾個公認的重要特性，理解它們能幫你判斷「你寫的到底算不算一個好演算法」：

- **input（輸入）**：演算法可以接收零個或多個輸入。例如「排序一串數字」這個演算法，輸入就是那串數字。
- **output（輸出）**：演算法至少要產生一個輸出，也就是你想要的結果。沒有輸出的步驟序列沒有意義。
- **definiteness（明確性）**：每一個步驟都必須精確、沒有歧義。「加一點鹽」對食譜也許還行，但對電腦來說「一點」是多少？電腦需要的是「加 5 公克鹽」這種毫無模糊空間的指令。
- **finiteness（有限性）**：演算法必須在有限的步驟之後**終止**。一個永遠跑不完的程序，嚴格來說不算是演算法。這也是為什麼寫出無限迴圈（infinite loop）通常是個 bug。
- **effectiveness（可行性）**：每一步都必須是能被實際執行的基本操作，在有限的時間與空間內完成。

很重要的一點：**演算法獨立於任何程式語言而存在**。同一個「找出陣列中最大值」的演算法，你可以用 JavaScript 寫、用 Python 寫、用畫流程圖（flowchart）表達，甚至用自然語言口述。程式語言只是把演算法「翻譯」成電腦看得懂的形式的一種載體。這個觀念很關鍵，因為它意味著：**先想清楚解法（演算法），再決定怎麼把它寫成程式碼**，而不是一邊打字一邊瞎猜。

順帶一提，演算法和 **heuristic（啟發法）** 不同。演算法透過精確的程序**保證**得到正確答案；啟發法則是在無法或不值得求精確解時，用經驗法則給出「夠好」的近似答案，不保證最佳。社群媒體的推薦系統就比較偏啟發法——它不保證推給你的是「最完美」的貼文，只是根據一堆訊號猜一個還不錯的。

### 為什麼 web 開發者要懂演算法

你可能會想：我只是要做網站，這些跟我有關嗎？有的，而且關係比你想的更直接。

第一，**效能**。當使用者數量或資料量放大，「怎麼解」會直接決定你的網站是秒開還是轉圈圈。一個 O(n²) 的做法在 100 筆資料時毫無感覺，在 100 萬筆時可能就是幾分鐘的差別。懂一點演算法，你才知道哪裡藏著效能陷阱、該換什麼策略。

第二，**別重複造輪子**。很多常見問題——排序、搜尋、去重、找最短路徑——前人早就研究出成熟又高效的解法。你不需要自己發明，但你需要**認得出**「這其實是某個經典問題」，然後套用對的工具。這正是學 CS 概念的價值：它給你一套辨識問題型態的詞彙。

第三，**面試**。這門課程的目標是讓你為職涯做好準備。幾乎所有軟體工程的技術面試，都會直接考 **DSA（Data Structures and Algorithms，資料結構與演算法）**。公司之所以愛考這個，是因為它能快速看出一個候選人「面對沒看過的問題時，怎麼拆解、怎麼思考、怎麼權衡取捨」——這比背了多少 API 更能反映一個工程師的底子。所以練 DSA 不只是為了過關，更是在鍛鍊你的問題解決肌肉。

### 什麼是 pseudocode（虛擬碼）

在真正動手寫程式之前，聰明的工程師常常會先寫 **pseudocode（虛擬碼）**。虛擬碼是「用混合了程式語言慣例與自然語言的方式，描述一個演算法的步驟」。它介於「人話」和「程式碼」之間：比純文字更有結構，但又不受任何語言的語法規則束縛。

為什麼要多這一道手續？

- **先想邏輯，別分心於語法**：寫虛擬碼時你不用煩惱分號、括號、變數宣告該怎麼寫，可以把腦力全部集中在「這個問題到底該怎麼解」。等邏輯理順了，再翻譯成真正的 JavaScript，過程會順很多。這是一種由上而下（top-down）的思考方式。
- **跨語言、跨程度都看得懂**：虛擬碼不綁定特定語言，所以無論讀者用什麼語言、程度深淺，都能理解演算法的核心原理。它是描述「關鍵想法」而不依賴特定環境的最佳工具。
- **當你的思考草稿**：面對一個複雜、陌生的演算法時，先用虛擬碼把它拆成一步步，能大幅降低出錯的機率。

虛擬碼沒有唯一標準寫法。常見慣例包括：省略變數宣告、用接近自然語言的句子取代細節、借用像 `IF`、`ELSE`、`WHILE`、`FOR`、`RETURN` 這類關鍵字來表達控制流程。有人寫得很像真正的程式碼，也有人寫得比較像條列式的散文，兩種都可以，重點是**清楚、無歧義**。唯一要避免的是在同一份虛擬碼裡混用太多種語言風格，那反而會讓人看不懂。

要記得虛擬碼的一個本質限制：它**不能被電腦執行**，只能靠人腦逐行檢查。它是設計階段的思考工具，不是最終產物。

### 把觀念串起來

整理一下這一課的核心脈絡：面對一個問題，好的工程師不會一開始就埋頭打字。他會先在腦中（或紙上）設計一個 **algorithm**——一組有限、明確、能終止並產生正確輸出的步驟。為了把這個演算法想清楚、講清楚，他會先寫成 **pseudocode**，暫時擺脫語法細節，專注在邏輯本身。等邏輯確認無誤，才把虛擬碼翻譯成真正的 JavaScript。這套「先設計、再實作」的習慣，加上對常見演算法與資料結構的熟悉，就是把問題「解出來」升級成「解得好」的關鍵，也是技術面試最想看到的能力。

## 程式碼範例

下面用一個經典的小問題示範「從 pseudocode 到 JavaScript」的流程：**找出一個陣列中的最大值**。

先寫虛擬碼，把邏輯想清楚（這一段不是任何語言，是給人看的）：

```text
演算法：找出陣列中的最大值
輸入：一串數字 numbers
輸出：其中最大的數字

1. 假設第一個數字 numbers[0] 就是目前的最大值 max
2. 從第二個數字開始，逐一走訪陣列裡的每個數字 current
3.   如果 current 比 max 還大
4.     就把 max 更新成 current
5. 走訪完畢後，回傳 max
```

接著把同一套邏輯翻譯成可執行的 JavaScript：

```js
// 找出陣列中的最大值
function findMax(numbers) {
  // 對應虛擬碼第 1 步：先假設第一個元素是最大值
  let max = numbers[0];

  // 對應第 2 步：從索引 1 開始走訪剩下的元素
  for (let i = 1; i < numbers.length; i++) {
    const current = numbers[i];
    // 對應第 3、4 步：發現更大的就更新 max
    if (current > max) {
      max = current;
    }
  }

  // 對應第 5 步：回傳最終結果
  return max;
}

console.log(findMax([3, 7, 2, 9, 4])); // 9
```

注意程式碼裡的每一段註解，幾乎都能對回虛擬碼的某一步。這就是虛擬碼的價值：它是你的施工藍圖，翻譯成程式碼時幾乎不需要再重新思考邏輯。

同一個問題，其實也有更「JavaScript 風格」的優雅解，直接用內建方法一行搞定：

```js
// 利用展開語法（spread）搭配 Math.max
function findMax(numbers) {
  return Math.max(...numbers);
}

console.log(findMax([3, 7, 2, 9, 4])); // 9
```

這正好呼應了核心概念：**同一個演算法問題，往往有暴力（手寫迴圈）與優雅（善用既有工具）兩種解法**。認得出「這是找最大值」這個經典問題，你才會想到去用 `Math.max` 這個現成的輪子。

## 常見陷阱

!!! warning "把「會寫程式碼」誤當成「懂演算法」"
    很多初學者一拿到問題就急著打開編輯器開始打字，靠不斷試誤（trial and error）湊出能跑的結果。這在小問題上勉強行得通，但一遇到稍微複雜的題目、或是面試現場，就會卡死。正確的順序是：**先用 pseudocode 把演算法想清楚，確認它有明確步驟、會終止、能產出正確輸出，再動手寫程式碼**。程式語言只是把想法翻譯出來的工具，真正值錢的是想法本身。另外也提醒：虛擬碼是設計階段的思考草稿，它**無法被電腦執行**，別把它跟能跑的程式碼搞混。

## 練習

這一課沒有需要動手寫的專案，重點在建立正確的觀念與心態。請依序完成以下反思步驟（原文以影片與文章為主，以下已將重點濃縮，你不點連結也能掌握）：

1. **理解「如何思考演算法」**：把演算法想成一份「解決某類問題的精確食譜」。練習拿一件日常任務（例如泡一杯茶、從家裡走到最近的便利商店），試著把它拆解成一步步明確、不含糊、會結束的指令。
2. **弄懂「什麼是演算法」的正式定義**：一個演算法要具備 input、output、definiteness（明確）、finiteness（有限會終止）、effectiveness（每步可執行）這幾個特性。回頭檢查你上一步寫的步驟，是否每一項都符合。
3. **想清楚「演算法對 web 開發為什麼重要」**：關鍵在於**效能**（資料量放大時，解法好壞決定網站快慢）與**不重複造輪子**（認得出經典問題，就能套用成熟解法）。
4. **練習寫 pseudocode**：挑一個你熟悉的小功能，先不要寫真正的程式碼，改用虛擬碼描述它的邏輯，借用 `IF`、`WHILE`、`RETURN` 等關鍵字，專注在邏輯而非語法。
5. **認識 DSA 與求職的關聯**：了解為什麼幾乎所有軟體工程面試都會考 **DSA（資料結構與演算法）**——公司想看的是你面對陌生問題時的拆解與取捨能力，而不只是你背了多少語法。

完成後，試著不看講義回答以下三個問題，檢驗自己是否真的懂了：

- 什麼是 algorithm（演算法）？它有哪些必備特性？
- 演算法對 web 開發為什麼重要？
- 什麼是 pseudocode（虛擬碼）？為什麼要在寫程式前先寫它？
