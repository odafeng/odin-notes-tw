# 面試準備與應對

> 改寫自 The Odin Project：[Preparing to Interview and Interviewing](https://www.theodinproject.com/lessons/node-path-getting-hired-preparing-to-interview-and-interviewing)
> ｜Full Stack JavaScript › Getting Hired › Applying to and Interviewing for Jobs

## 核心概念

對多數人來說，面試（interview）和公開演講並列為最恐懼的事情之一。你不只是在別人面前表演，對方還會全程主動評判你。但與其被這股壓力壓垮，不如換個角度：把面試當成一個「炫技」的機會，去展示你親手打造的作品，以及一路學到的有趣技能。最好的面試，其實是一場帶有技術深度、雙方都很投入的對話。

### 準備是第一步

在所有事情之前，你要先做準備。準備分兩大塊：

- **預想問題與答案**：列出你可能會被問到的問題，並為每一題想好能展現你實力的回應方向。不用背逐字稿，但要對「怎麼講」有把握。
- **研究公司**：了解這家公司的產品、技術與痛點。這能讓你把自我推銷（pitch）調整成貼近他們的需求，也能在被反問「你有什麼問題想問」時，提出有見地的問題。

一個實用小技巧：對著鏡子練習談論自己，不要害羞。愈熟悉「介紹自己」這件事，正式上場時就愈自然。

### 招募流程長什麼樣

一家典型科技公司招募開發者，通常會走過以下漏斗（funnel）。了解每一關的目的，你就知道每一關該端出什麼：

1. **Phone Screen（電話初篩）**
2. **Technical Interview（技術面試）**
3. **Technical Challenge（技術挑戰／帶回家作業）**
4. **Fit Interviews（文化契合面試）**
5. **Job Offer（錄取通知）**
6. **Offer Negotiation（條件協商）**
7. **Offer Acceptance（接受錄取）**

注意順序不是鐵律。有些公司會把技術挑戰放在最前面，有些會直接跳過輕鬆的初篩、劈頭就進技術測驗，所以每一關都要有心理準備。

### 電話初篩（Phone Screen）

恭喜，你的履歷沒被直接刷掉。電話初篩通常是半小時左右，對象往往是人資（HR），而不是實際的用人主管（hiring manager）。它的真正目的，是先確認你有相當機會通過後面的技術面試與文化面試，可以把它想成後面每一關的「輕量版」。

你大概會被問到履歷上列的一些技術，但不會深挖細節（雖然有些公司會塞一道腦筋急轉彎），也會被問一些比較「軟」的問題，例如你為什麼選這份工作、你以前做過什麼。各家公司用電話初篩的方式差異很大。這一關沒有什麼花招，真正的策略就是「不要有策略」：誠實、有熱情、坦率。

### 技術面試（Tech Interview）

技術面試通常是整個流程中最令人害怕的一關，因為這是他們評估你「到底有沒有技術底子」的地方。你不只會被具體地問到你做過的工作，還可能被要求當場解邏輯題、即時寫程式（live coding），或是在白板上設計新功能。

這一關有個常被誤解的重點：**面試官常常故意把你逼到極限，就是想看你「不會」的時候會怎麼反應。**如果一題你解得太輕鬆，他們會換更難的。尤其身為新手，你一定會卡關，這很正常。你最大的資產是誠實與求知欲。

幾個具體的應對原則：

- **邊做邊講**：解題時保持清晰、有邏輯，把每一步「為什麼這樣做」大聲說出來。遇到障礙就把它講出來，並說明你在「真實世界」會怎麼找解法。很多時候答案就是「上網 Google 某個函式」，那就大方說出來！面試官知道你不是 Ruby／JavaScript 專家，但他們需要確認你有能力靠自己找到解法，因為工作上一定會遇到不會的東西。
- **先用暴力解也沒關係**：用效率不高的暴力法（brute force）起手，往往是最好的開始，能確保你抓對了題目的感覺。面試官通常會接著問「怎麼讓它更好」，這遠比你一心想寫出天才解、最後時間耗盡卻交白卷要好得多。你的任務不是要「聰明絕頂」，而是要在挑戰面前展現「靈活與縝密」。
- **不會就說不會**：卡住時，主動說出來、和面試官一起想辦法，遠比默默硬撐好。對面試官來說，最難受的就是看著應徵者在題目上愈來愈挫折、卻不開口求助、也不透露任何思路。他們其實跟你一樣希望你成功。

此外，你需要補讀一些前面課程沒有著重的東西，特別是**資料結構（data structures）與演算法（algorithms）**，因為它們是面試官最愛的考點。它們也許不是衡量寫程式能力的好指標，但業界慣例如此，這類偏「電腦科學」的題目你一定會遇到。

### 技術挑戰（Take-home Tech Challenge）

帶回家的技術挑戰可能排在現場面試之前或之後，看公司安排。他們會給你一個應用程式或問題，讓你用自己的時間完成，通常最多花上一整天。常見的形式包括：寫一個附測試（tests）的範例 web app，或用程式碼解一道有難度的演算法問題。

評分重點是**解答的完整度**與**程式碼品質**。如果這關被放在技術面試之前，它同時也是一種「承諾度」測試，因為多達一半的人根本不會交出解答。所以光是「認真完成並繳交」，你就已經贏過很多人。

### 最後一關：文化契合（Fit）

最終決定前的最後一步，通常是到對方辦公室、花好幾個小時見完整個團隊，並接受一輪操練。他們可能還是會考你技術，但真正的目標是確認「你是不是一個好共事的人」。這一關往往是團隊共識制：只要有任何一位成員覺得你不合適，通常你就不會被錄取。建議？就算你在家裡再怎麼放飛，面試時別表現得太怪或太尷尬就好。

反過來說，這也是**你評估他們**的機會。能走到這一步，代表雙方很可能是不錯的配對。你要判斷這是不是你想效力的公司，所以請帶一份問題清單，確保它們都得到解答。

### 談薪水（Compensation）

被問期望待遇時要謹慎。公司幾乎一定會問你「期望薪資多少」，而且常常在電話初篩就問。**給一個單一具體數字往往有害無益**：報得比他們的範圍高，你可能被視為太貴；報得太低，他們就知道可以把你的 offer 壓在比原本更低的數字。比較安全的做法是**給一個區間**（例如「我期望落在 80,000 到 90,000 美元之間」），既能貼近他們的預算，也替後續協商留下空間。

拿到 offer 之後，可以透過詢問業界的人，或用薪資比較網站（如 Levels.fyi、Glassdoor、Payscale 等）來核對是否合乎行情。「junior」不等於「被低薪」，但記得依你的年資（0～1 年）調整，才能得到最準確的參考。

此外，如果是招募人員（recruiter）主動來找你，事先禮貌地詢問這個職缺的預算，也可能很有幫助，但務必得體。可以用「這個職缺有沒有一個規劃好的薪資範圍？」這樣的方式開口。有時對方會回「薪資依經驗而定」，那就把你的期望區間放在心裡，隨著面試流程走下去。

## 程式碼範例

本課以觀念為主，沒有需要執行的程式碼。這裡改為整理幾個「值得放進待辦清單」的準備動作，供你在面試前逐項打勾：

```text
面試準備檢查清單（Interview Prep Checklist）
────────────────────────────────────────
[ ] 研究公司：產品、技術棧、近期動態、痛點
[ ] 準備 3～5 個要反問面試官的好問題
[ ] 對著鏡子練習「自我介紹」與「為什麼想加入」
[ ] 複習資料結構（陣列、鏈結串列、樹、雜湊表）
[ ] 複習常見演算法（排序、搜尋、DFS/BFS）
[ ] 練習「邊寫邊講」：口述你的思路與取捨
[ ] 想好一個薪資「區間」，而不是單一數字
[ ] 帶回家作業：預留完整時段、附上測試、務必繳交
```

## 常見陷阱

!!! warning "把 LeetCode 當成「無腦硬刷」"
    網路上流行一句話：「去刷 LeetCode 就對了。」但常見演算法通常有一個「訣竅」或固定步驟。準備面試時，花好幾個小時甚至好幾天硬磕一道題，收穫其實很小。更有效率的做法是：**先去查那道題背後的演算法／虛擬碼（pseudocode），理解它，再回頭練習需要用到該演算法的題目。**
    這就像要你算三角形的斜邊長，你卻完全不知道畢氏定理，那自然覺得不可能；一旦知道「訣竅」，計算就變得輕鬆。同理，若一題需要用到「樹的深度優先搜尋（Depth First Search）」，而你對 DFS 毫無概念，坐在那裡枯想幾天也沒用，不如先去查常見的樹演算法，再回來解題。

!!! warning "在初篩就報一個死板的薪資數字"
    公司很愛在電話初篩就問期望待遇。報單一數字風險很高：太高被嫌貴、太低被壓價。永遠優先給一個「區間」，替協商保留彈性。

!!! warning "卡住時默默硬撐、不出聲"
    技術面試中最糟的表現不是「不會」，而是「不會又不說」。面試官無法讀你的思路，只會看到你愈來愈挫折。誠實說出卡點、主動求助、把找解法的方式講出來，反而是加分項。

## 練習

本課的 Knowledge Check（自我檢核）沒有需要動手的專案，請用以下問題自問自答，答不出來就回到上面對應段落複習。重點是理解，不需要背誦：

1. **電話初篩的目的是什麼？** —— 提示：把它想成後面每一關的「輕量版」，由誰主持、想確認什麼。
2. **在技術面試中，你可以用哪些策略取得成功？** —— 提示：邊做邊講、先用暴力解、不會就誠實說並一起想。
3. **該如何面對帶回家的技術挑戰？** —— 提示：評分看完整度與程式碼品質，務必繳交（一半的人不交）。
4. **該如何處理談薪水這件事？** —— 提示：給區間而非單一數字，事後再對照行情核對。

延伸練習（自由選擇）：找一位同儕（peer）做一次模擬面試，其中一人扮演面試官出題，另一人「邊寫邊講」解題；結束後互換角色。這能同時練習「口述思路」與「面對未知時的反應」。
