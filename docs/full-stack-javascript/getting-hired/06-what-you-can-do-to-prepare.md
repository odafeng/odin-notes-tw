# 你可以做的準備

> 改寫自 The Odin Project：[What You Can Do to Prepare](https://www.theodinproject.com/lessons/node-path-getting-hired-what-you-can-do-to-prepare)
> ｜Full Stack JavaScript › Getting Hired › Preparing for Your Job Search

## 核心概念

求職是一段可以「有結構、有策略」去走的路，而不是投一堆履歷後被動等回音。用系統化的方式面對它，關鍵在於「及早開始」，把幾件重要的基礎工作先處理好，這樣後面真正投遞、面試時，你才有本錢從容應對。這一課就是在講：在你正式開始找工作之前，有哪些準備動作可以現在就做。

### 先把技術練到夠強（Get good）

整個求職流程可能很磨人，但別讓它分散你「持續累積技術實力」的注意力，因為技術實力最終才是決定你能不能拿到 offer 的核心。這門課假設你已經走完前面的課程，並且在完成它們的過程中累積了足以稱得上「有能力（competence）」的成果。

如果還沒有到那個程度，你必須非常誠實、甚至有點狠地面對自己的弱點，即使那是像 algorithm（演算法）與 data structure（資料結構）這種棘手的主題，也要補起來。原因很簡單：你申請的職位需要這些能力，逃避弱點只會在面試時被打回原形。準備求職的第一步，其實是「確認自己真的準備好了」。

### 打造你的個人敘事（Craft a narrative）

很多人一想到要「推銷自己（selling yourself）」就渾身不對勁，但這正是求職中你必須做的事。好消息是，這不代表你要像油腔滑調的中古車推銷員一樣誇大其詞，它比你想的更細膩。

所謂推銷自己，真正的意思是「說一個關於你的故事」：你是誰、你怎麼走到今天這一步、以及為什麼你會持續成長。每個人都會說故事，所以請認真花心思，把你的背景（尤其如果你來自非傳統背景，例如轉職者、自學者）編織成一段有說服力、讓人無法忽視的「能力與動力之流」。

這個敘事不是造假，而是把你原本零散的經歷，整理成一條清楚、連貫、指向未來的線。當雇主問「跟我聊聊你自己」時，你要能自信地講出這個故事，而不是支支吾吾地流水帳。

### 建立作品集（Build a portfolio）

在整個學習過程中，你都應該持續「動手做東西」。你做過的專案，是展現技術能力、打動雇主的關鍵。雇主想看到的是：你能不能實際建構並交付（build and ship）程式碼，以及在面對工程上的取捨（trade-off）時，你會不會做出好決策。而這一切，都是從「聊你的作品集」這件事延伸出來的。

作品集還有一個較少被提及、卻很重要的作用：社會證明（social proof）。如果你參與過較大型或團隊的專案，那幾乎等於在證明「你曾經在某個地方工作過」。它讓雇主放心：既然別人能跟你合作，那我也可以。

說到團隊專案，參與 open source（開源）社群是一個很棒的方式，既能展示技能，又能跟其他開發者協作。如果你在找第一個可以貢獻的開源專案，可以從 TOP（The Odin Project）本身開始，它的課程、主網站與工具都持續在改進，歡迎透過 GitHub 上的 repository（程式庫）參與。開源貢獻在求職時也能成為你敘事裡有力的一環。

最後要牢記：身為開發者，你的作品集就是 GitHub。你也許有個人網站，但 GitHub 依然是證明你技術能力最具權威性的地方。忽視它，代價自負。

### 整理你的網路形象（Clean up your web presence）

先做一個小實驗：開一個無痕視窗（incognito window，避開 Google 依你習慣客製化的演算法），搜尋你自己的名字。跳出來的是什麼？這些連結符合你想呈現的個人敘事嗎？

要知道，任何想僱用你的人，幾乎一定會 Google 你。所以你要確保他們找到的內容，正是你希望他們看到的。如果你的名字非常菜市場（很常見），要建立網路形象會比較辛苦，但反過來說，你也比較不容易出現「錯誤的」網路形象。

網路形象裡最該優先整理的，是你的 GitHub profile（個人檔案）。確保你的專案排列整齊，而且每個專案的 README 檔案都有清楚的描述，以及一個指向 live（線上運行版本）的連結，讓任何點進來看的人都能一眼看到你的厲害之處。

接著，努力讓自己「可被發現（discoverable）」。如果有人想聯絡你、或在網路上找你的作品，找得到嗎？你有個人網站嗎？你有在使用 X（前身為 Twitter）嗎？你不需要瘋狂經營社群媒體，但有時候光是有一個「像個真人」的檔案，給別人一個能聯絡到你的管道，就能帶來很大的差別。

如果你不排斥，也可以使用 LinkedIn。同樣地，這件事的重點是「用你選擇的方式，把你的個人敘事主動放到網路上」，而不是被動仰賴 Google 幫你拼湊。LinkedIn 其實正在快速取代傳統履歷的角色。此外像 BrandYourself 這類服務，會更細緻地教你如何經營網路形象，但核心原則不變：確保你的故事，是以你想要的方式被講述。

### 走進社群、蒐集資訊（Get out and collect information）

去認識其他開發者。這不是要你去「用力地拓展人脈（power networking）」，而是要建立真誠的人際連結。你的人脈網絡真的非常重要：現階段它也許只是幫你回答問題、收集建議，但你的下一份工作，很可能就是透過你認識的人、或你在蒐集建議過程中遇到的人牽線而來的。人們其實喜歡給建議，也喜歡在情感上投資那些「處在成長曲線上」的人。

最理想的形式是一對一的面對面交流，在那種場合你可以放心問一些「蠢問題」。但除此之外，還有很多社群管道可用，例如 Meetup、Stack Overflow、Quora、在地的使用者社團等等。加入它們，別害怕發問，提問本身就是建立關係的好方法。

你腦中大概有一百萬個問題，但在求職這個階段，最該問的是：各種工作「到底在做什麼」、以及「他們在找什麼樣的人」。最快的方法就是直接問開發者「你平常都在做什麼」，這比在部落格文章裡大海撈針快得多。必要時，請對方喝杯咖啡、吃頓午餐都值得。給自己設一個目標：找 5 個真實的人聊聊你的求職。

蒐集這類資訊還有另一招：大量翻閱職缺公告（job posting）。看多了，你會很快發現它們之間的共通點：重複出現的技能、常見的職稱、反覆被要求的條件。這工作很枯燥，但非做不可，因為它會校準你對「市場真正要什麼」的認知。

最後一點跟網路形象呼應：別害怕產出內容，例如寫部落格、或在 Quora 上回答問題。這些內容不一定要是純技術的，但應該能展現你「批判性思考」與「文字表達」的能力，而這兩項恰好都很得未來雇主的青睞。

### 小結

把這幾件事串起來看，它們其實圍繞同一個核心：在別人開始評估你之前，先主動把「你想被看見的樣子」準備好、擺出來。技術是地基，敘事是包裝，作品集是證據，網路形象是門面，社群人脈則是機會的來源。這些都不是投遞履歷當天才能臨時抱佛腳的，而是要及早、持續地累積。

## 程式碼範例

本課以求職觀念為主，沒有需要執行的程式碼。這裡改附一段「可直接照做」的行動清單，幫你把上面的觀念落地：

```text
# 求職準備行動清單（本週就能開始）

[ ] Get good：列出自己 3 個技術弱點，各排一個補強計畫（含演算法/資料結構）
[ ] 敘事：用 3～5 句話寫出「你是誰、怎麼走到這裡、為何會持續成長」
[ ] 作品集：挑 3 個代表作，替每個補上 README（描述 + live 連結）
[ ] GitHub：整理 profile，把代表作 pin（釘選）到首頁
[ ] 網路形象：用無痕視窗 Google 自己，記下不符合敘事的連結
[ ] 可被發現：確認至少有一個管道（個人網站 / LinkedIn / X）能聯絡到你
[ ] 社群：設定目標——找 5 個真人聊聊他們的工作與求職經驗
[ ] 情報：翻 20 則職缺，整理出反覆出現的技能與關鍵字
[ ] 內容：寫 1 篇部落格或回答 1 題技術問題，展現思考與表達
```

## 常見陷阱

!!! warning "把求職準備當成「等技術完美之後」才做的事"
    敘事、作品集整理、網路形象、社群人脈，這些都需要時間發酵，尤其人脈與內容更是。如果你打算等到「技術練到完美」才開始準備，往往會發現時間根本來不及。正確做法是「一邊持續補強技術，一邊及早鋪陳這些準備」，兩者並行，不是先後。

!!! warning "忽視 GitHub，只顧經營個人網站"
    很多人花大把時間做華麗的個人網站，卻放著 GitHub profile 亂七八糟。對雇主而言，GitHub 才是驗證你技術能力最權威的地方：雜亂的 repository、沒有 README、看不到 live 連結，會讓再漂亮的個人網站都打折扣。優先把 GitHub 整理好。

!!! warning "把「網路形象」誤解成要瘋狂經營社群"
    重點不是變成網紅或天天發文，而是讓你「像個真實、可聯絡到的人」，並確保搜尋到的內容符合你想呈現的敘事。與其追求聲量，不如先確保「找得到你」且「找到的是對的東西」。

## 練習

以下把原課程的 Assignment 改寫成繁中步驟。這些多為閱讀性資源，建議照著讀完並做筆記：

1. **如果你正在轉職**：閱讀 Dorie Clark 的〈How to Explain Your Career Transition〉，學習如何把「職涯轉換」講成一個有說服力、而非需要辯解的故事。
2. **理解 GitHub 為何如此重要**：閱讀以下三篇，體會「直接展示真實程式碼」為何比履歷更有力——
   - 〈A Programmer's Portfolio〉（Coding Horror）：為何秀出實際程式碼比履歷更能證明實力。
   - 〈The Power of a Programming Portfolio〉（grokcode）：作品集如何幫你脫穎而出。
   - 〈Your GitHub Portfolio〉（Forbes）：GitHub 如何成為雇主找人才、看程式碼的核心工具。
3. **學習如何「經營個人品牌」**：閱讀關於自我品牌與線上聲譽管理的文章（如 BrandYourself、Monster 的〈Control Your Online Reputation〉、Stack Exchange 上關於「開發者是否需要線上形象」的討論，以及 Pramp 的〈How to Build Your Digital Presence as a Software Developer〉）。
4. **閱讀經典《How to Win Friends and Influence People》（人性的弱點）**：這本書談的是商業與人際，適用於你身邊社交生活的所有面向。若沒時間讀整本，可先看重點摘要。
