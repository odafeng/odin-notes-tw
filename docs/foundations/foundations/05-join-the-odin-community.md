# 加入 Odin 社群

> 改寫自 The Odin Project：[Join the Odin Community](https://www.theodinproject.com/lessons/foundations-join-the-odin-community)
> ｜Foundations › Introduction

## 核心概念

寫程式常被想像成一個人對著螢幕孤軍奮戰，但真實的軟體開發是一件高度協作的事。學會跟別人一起工作、互相求助與互相幫忙，本身就是 web developer 的核心能力之一。The Odin Project 有一個活躍的 Discord 社群，這一課的目標就是讓你把「加入社群」當成學習流程的一部分，而不是可有可無的附加品。

### 社群對你的價值

自學這條路又長又辛苦，途中一定會遇到程式碼跑不動、甚至連自己在寫什麼都看不懂的時刻（The Odin Project 把這種狀態稱為「絕望沙漠」，desert of despair）。在社群裡，永遠有走在你前面幾步的人願意拉你一把；同時，當你回頭去幫走在你後面的人時，為了把觀念講清楚，你反而會把自己學到的東西鞏固得更深。教別人，其實是最好的複習。

此外，當你辛苦做完一個 project、成就感滿滿時，社群裡的人最能體會那份努力的份量，是分享成果最好的地方。

### 開口求助之前

社群很棒，但不代表你一遇到問題就該立刻丟出來問。很多時候，卡關的不是你的程式碼，而是你的**思路（approach）**。開口之前，先做這幾件事：

- **先停下來喘口氣。** 把大問題拆成小塊，找出真正卡住你的那一塊到底是什麼。
- **試試小鴨除錯法（rubber duck debugging）。** 對著一隻橡皮小鴨（或任何東西）一行一行把程式碼講出來，逼自己解釋每一步在做什麼。很多 bug 會在你「講給別人聽」的過程中自己現形。
- **善用搜尋引擎。** 把錯誤訊息或關鍵字丟去 Google，通常早有人遇過同樣的問題。回頭翻先前的課程，也可能找到能套用的工具。

只有在這些方法都試過、仍然無解時，才是求助社群的時機。

### 怎麼問出一個好問題

丟出問題前，先記住一句社群格言：**別「問能不能問」（don't ask to ask）**。與其問「有人懂 CSS 嗎？」然後乾等，不如直接把問題完整講出來——這樣別人一看就知道能不能幫、也更願意伸出援手，你也能更快得到答案。

一個好問題應該附上足夠的**脈絡（context）**：

- 你認為問題出在哪裡？
- 你**希望**發生什麼結果？
- **實際上**發生了什麼？
- 你是怎麼走到這一步的？
- 你已經試過哪些方法？

如果講不清楚問題，可以附上**截圖（screenshot）**，這對展示 command line（命令列）的輸出或錯誤訊息特別有用。但截圖不能取代程式碼——請同時把相關的程式碼一起貼出來（用正確格式），或把 project 推上 GitHub、或用線上 REPL（例如 CodePen）分享，讓別人能實際檢視並重現問題。光有截圖、沒有可執行的程式碼，別人很難幫你 debug。

如果一時沒人回應，別急。先善用 Discord 的**搜尋功能**，用關鍵字或錯誤訊息查查看，說不定別人早就問過、也解決過了。

### 把問題排版好

清楚的排版能讓所有人更容易讀懂你的問題。在聊天室裡，用**反引號（backtick，`` ` ``）**可以把程式碼跟一般句子區分開來。反引號在美式與英式鍵盤上位於 Tab 鍵上方，和位於 Enter 鍵左邊的單引號（`'`）不一樣，別搞混。

- **單行程式碼：** 在程式碼前後各放一個反引號。
- **多行程式碼：** 在程式碼上下各放三個反引號（各自獨立一行）。
- 想要語法上色，可以在開頭的三個反引號後面標明語言，例如 `js` 或 `javascript`。

### 反過來，怎麼幫助別人

會問問題只是一半，會**幫別人**同樣重要。The Odin Project 為社群互助整理了一套原則，核心精神是：幫忙是為了讓對方成長，而不是替他把答案寫完。幾個重點：

- **引導，而不是直接給答案。** 除非只是明顯的 typo（打字錯誤）或語法錯誤，否則用提問引導對方自己找到答案，例如「你已經試過什麼？」「你預期這個 function 會做什麼？」「你覺得那個錯誤訊息是什麼意思？」這樣才能真正培養對方的除錯能力。
- **只在你有十足把握時才回答。** 沒有 100% 確定就別亂答，錯誤的答案往往比沒有答案更糟。好的答案值得等待。
- **別人正在幫忙時，就別插隊。** 同時有好幾個人回話，只會讓求助者更混亂。
- **先解決原本的問題，再談其他。** 就算你一眼看到別的毛病，也先把對方問的那題解決掉。
- **不回答用 Google 就能查到、或課程裡已經教過的問題。** 這不是冷漠，而是幫對方養成自己研究的能力。
- **鼓勵對方善用除錯器（debugger）、適時退一步休息，也坦承自己不會的地方。** 遇到超出你能力範圍的問題，誠實說「我不確定」並讓更有經驗的人接手，比硬撐更負責任。
- **保持耐心，情緒上來就先離開。** 你是志工，沒有義務忍受失控的對話；禮貌地退出、讓別人接手就好。

這些原則現在讀過有個印象即可，等你之後開始回頭幫別人時，再回來複習一次。

## 程式碼範例

本課以觀念為主，實際會用到的是 Discord 聊天室的格式化技巧與指令：

```text
單行程式碼：用一個反引號前後包住
`const name = "Odin";`

多行程式碼：用三個反引號，各自獨立一行；開頭可標語言以語法上色
```js
function greet() {
  console.log("Hello, Odinite!");
}
```

常用聊天指令：
/gif hi        送出一張打招呼的 giphy
/              查看所有可用指令
@username ++   對幫助你的人表達感謝
```

## 常見陷阱

!!! warning "只丟截圖、不給程式碼"
    截圖很適合展示錯誤訊息或畫面呈現，但別人無法在截圖上動手 debug。務必同時附上可重現問題的程式碼——貼進聊天室（用正確格式）、推上 GitHub，或用 CodePen 之類的線上 REPL 分享。缺了可執行的程式碼，幫你的人只能猜。

!!! warning "問「能不能問」然後乾等"
    別只丟一句「有人會 JavaScript 嗎？」就等回應（don't ask to ask）。直接把完整問題與脈絡一次講清楚，別人才判斷得出能不能幫你，你也能更快得到答案。

## 練習

1. 先[申請一個免費的 GitHub 帳號](https://github.com/join)。GitHub 是開發流程中不可或缺的一環，你很快就會用到它。
2. 登入 [Odin 官方 Discord 伺服器](https://discord.gg/fbFCkYabZB)，進去打聲招呼！伺服器裡有一間 introductions（自我介紹）房間，很適合介紹自己；每個課程主題也都有對應的聊天室，登入後四處逛逛吧。
   - **（選做）把 GitHub 連到你的 Discord 個人檔案：** 點使用者名稱右邊的齒輪圖示開啟 `User Settings`，進到 `Connections`，點 GitHub 圖示，在新分頁按「Allow Access」，再回到 Discord 確認「Display on profile」有打開。這樣別人就能看到你在做什麼，你也能看到別人。
3. 開始互動前，先記住這些準則：
   - **讀過規則與 FAQ：** 在 Discord 左側欄找到 `TOP META` → `rules` 和 `faq`，花點時間讀懂。
   - **記得螢幕另一端是真人：** 每個帳號背後都是有感受的人，請保持善意；沒好話說就別說。
   - **不會當面說出口的話，就別打出來。**
   - **@ 人要有目的：** 只在必要時 @ 對方，並把問題或訊息一起附上，等對方回覆再繼續 ping。
   - **不要洗版：** 把整段話打完再一次送出，別一句一句連發。
   - **不要排擠任何人：** 這些是公開聊天室，有人加入話題就把他納進來（例外是一對一協助學習者時，維持 1:1 才不會讓對方混亂）。
   - **問完程式問題別馬上消失：** 貼出問題後，請留下來跟幫你的人討論。
   - **先觀察再加入：** 花點時間感受社群怎麼互動，再跳進去。
